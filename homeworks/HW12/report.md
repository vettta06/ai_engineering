# HW12 – временные ряды: temporal split, baseline-модели и GRU-прогноз

## 1. Кратко: что сделано

- Какой датасет использовался и что именно прогнозировалось.  Использовался датасет S12-hw-dataset.csv (4320 наблюдений, часовая частота, диапазон дат: 2025-01-01 — 2025-06-29); прогнозировалась числовая колонка target на один шаг вперёд (горизонт прогноза = 1 час).
- Как был организован `train / validation / test` по времени.
Выполнен корректный temporal split без перемешивания данных: первые 70% наблюдений (3024 строки) — train, следующие 15% (648 строк) — validation, последние 15% (648 строк) — test; границы фиксированы по хронологии, random split не применялся.
- Какие подходы сравнивались (`B1`, `B2`, `B3`, `R1`).
B1 (naive-last): прогноз = последнее известное значение target из train;
B2 (moving-average): прогноз = скользящее среднее последних 7 значений с рекурсивным обновлением;
B3 (ridge-lag-features): Ridge-регрессия (alpha=1.0) на лаговых (lag_1, lag_7, lag_14), rolling- (rolling_mean_7, rolling_std_7) и календарных признаках;
R1 (gru-forecast): GRU-сеть (hidden_size=64, num_layers=2, dropout=0.2) на последовательностях длиной window_size=24 с использованием тех же признаков.
- Какой подход оказался лучшим и по какой причине.
Лучшим оказался подход R1 (GRU) с итоговой метрикой Test MAE = 5.73; GRU превзошла лучший baseline (B3, Test MAE ≈ 7.1) на ~19% благодаря способности моделировать нелинейные временные зависимости, улавливать долгосрочные паттерны через скрытое состояние и адаптироваться к изменяющейся динамике ряда (тренд + сезонность), что недоступно линейным методам и простым эвристикам.

## 4. Temporal split и признаки

### 4.1. Разбиение по времени

- Как именно выполнен `train / validation / test`: последовательное разбиение без перемешивания, первые 70% — train, следующие 15% — validation, последние 15% — test.
- Какие доли или интервалы использовались: 70% / 15% / 15% (3024 / 648 / 648 наблюдений)
- Почему `random split` здесь некорректен:
Нарушает временную зависимость (автокорреляцию)
Приводит к data leakage (модель видит будущее при обучении)
Даёт оптимистичные оценки качества
Не соответствует реальному сценарию прогнозирования

### 4.2. Признаки для baseline-моделей

- Какие лаги использовались: lag_1, lag_7, lag_14
- Какие rolling-признаки использовались: rolling_mean_7, rolling_std_7 (окно 7 часов, со сдвигом на 1 для избежания утечек)
- Какие календарные признаки использовались: hour, day_of_week, is_weekend, hour_sin, hour_cos, dow_sin, dow_cos
- Как обрабатывались пропуски после построения лагов/rolling: заполнены fillna(0) для GRU, удалены для Ridge
- Использовалось ли масштабирование и где именно оно обучалось: StandardScaler, обучен ТОЛЬКО на train, применён к val и test без переобучения

## 5. Модели и эксперименты (B1, B2, B3, R1)

Опишите коротко и сопоставимо:

- B1 (`naive-last`): прогноз = последнее известное значение target из train. Без признаков, без обучения.
- B2 (`moving-average`): прогноз = скользящее среднее последних 7 значений. Рекурсивное обновление на validation.
- B3 (`ridge-lag-features`): Ridge регрессия (alpha=1.0) на 12 признаках (лаги, rolling, календарные). StandardScaler на train.
- R1 (`gru-forecast`): GRU (hidden=64, layers=2, dropout=0.2) на последовательностях длиной 24 шага. Adam optimizer, lr=0.001, early stopping.

Дополнительно:

- Основная метрика выбора лучшей модели: best_val_mae
- Дополнительные метрики: RMSE, MAPE
- `window_size` для `GRU`: 24 часа
- Batch size: 32
- Максимальное число эпох: 50 (фактически обучено 26, early stopping)
- Optimizer: Adam
- Learning rate: 0.001 (с scheduler ReduceLROnPlateau)
- Как сохранялась лучшая модель: по минимуму validation MAE, сохранён state_dict()

## 7. Анализ

(8-15 предложений)

Нужно прокомментировать:

- почему наивный baseline оказался слабым или, наоборот, surprisingly сильным;
- что дали `moving average` и лаговые признаки;
- почему `Ridge` на признаках сработала лучше или хуже простых baseline;
- почему `GRU` помогла или не помогла на данном ряде;
- насколько заметны тренд/сезонность и как это отражается на прогнозе;
- где в такой задаче могли бы появиться утечки данных;
- почему выбор по `validation`, а не по `test`, принципиален;
- какие ошибки модели оказались наиболее показательными.
Наивный baseline (B1) оказался самым слабым (MAE=20.34), что ожидаемо для ряда с трендом и сезонностью — последнее значение не отражает структуру данных. Moving average (B2) улучшил результат на 38% за счёт усреднения шума, но всё ещё не улавливал сложные паттерны. Ridge на лаговых признаках (B3) показал значительный прогресс (MAE=6.53), так как явно использовал автокорреляцию и календарные эффекты. GRU превзошла все baseline (Val MAE=4.89, Test MAE=5.73) благодаря способности моделировать нелинейные временные зависимости и долгосрочные паттерны через скрытое состояние. Тренд и сезонность хорошо видны на графиках — GRU успешно их воспроизводит, хотя на пиках есть небольшие отклонения. Утечки данных могли появиться при неправильном построении лагов (например, shift(-1) вместо shift(1)) или при масштабировании на всём датасете вместо только train. Выбор по validation, а не по test, принципиален — test используется один раз для финальной оценки, иначе возникает переобучение на test и оптимистичные оценки. Наиболее показательные ошибки модели — на экстремальных значениях (выбросах), где GRU склонна к консервативным прогнозам.

## 8. Итоговый вывод

(3-7 предложений)

- Какой подход вы бы взяли как базовый для такой задачи и почему.
- В каком случае достаточно простого baseline на признаках.
- В каком случае имеет смысл переходить к `GRU`/другим нейросетям.
- Что главное вы поняли про корректную валидацию временных рядов.
Для данной задачи я бы взял GRU как базовый подход — она показала наилучшее качество и способна улавливать сложные временные зависимости. Простого baseline на признаках (Ridge) достаточно, когда данных мало, вычислительные ресурсы ограничены, или ряд имеет простую линейную структуру. Переход к GRU/нейросетям имеет смысл при большом объёме данных, наличии нелинейных паттернов, и когда критично качество прогноза. Главное, что я понял про валидацию временных рядов — random split недопустим, нужен temporal split с чётким разделением train/validation/test по времени, и все признаки должны строиться без утечки из будущего.
