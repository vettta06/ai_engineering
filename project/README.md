Проект посвящён созданию сервиса автоматической оценки кредитного риска на основе табличных данных заёмщика.

## 2. Структура проекта

Проект организован в следующей структуре:

- `requirements.txt` – зависимости проекта (библиотеки Python, необходимые для запуска).
- `report.md` – отчёт по проекту (постановка задачи, данные, эксперименты, результаты).
- `self-checklist.md` – чеклист самопроверки проекта перед сдачей.
- `notebooks/` – экспериментальные ноутбуки:
  - EDA, прототипы моделей, отдельные эксперименты.
  - `01_eda.ipynb` – разведочный анализ данных;
  - `02_baseline_models.ipynb` – обучение, кросс-валидация и сравнение моделей;
  - `03_model_packaging.ipynb` – упаковка модели и тестирование пайплайна.
- `src/` – основной код проекта:
  - модули для работы с данными;
  - обучение моделей;
  - запуск сервиса/скриптов.
  - `main.py` – точка входа и конфигурация приложения;
  - `endpoints.py` – маршруты `/health`, `/predict`, `/metrics`;
  - `dependencies.py` – сервис загрузки модели и зависимостей;
  - `schemas.py` – схемы валидации запросов/ответов (Pydantic).
- `data/` – демонстрационные/учебные данные:
  - **только открытые, синтетические или обезличенные данные;**
  - при необходимости – небольшие выборки для быстрого запуска.
- `configs/` – конфигурационные файлы:
  - `model_config.json`, `app_config.json`
- `tests/` – тесты (`test_api.py`, `test_data.py`, `conftest.py`).
- `artifacts/` – сохранённые модели, отчёты, результаты экспериментов.
- `.github/workflows/` – конфигурация CI/CD (GitHub Actions).
- `Dockerfile` и `docker-compose.yml` – файлы для контейнеризации сервиса.

Структура полностью соответствует шаблону курса, дополнена файлами контейнеризации и CI/CD.

## 3. Требования и установка

### 3.1. Требования

- Python `>= 3.10` (уточните при необходимости).
- Установленные системные зависимости (если есть: например, `git`, `make` и т.п.).

### 3.2. Установка окружения

Примерный сценарий (можно адаптировать под ваш проект):

```bash
# Перейти в папку проекта
cd project

# Создать виртуальное окружение (опционально, но рекомендуется)
python -m venv .venv

# Активировать окружение:
# Windows:
.venv\Scripts\activate
# Linux / macOS:
source .venv/bin/activate

# Установить зависимости
pip install --upgrade pip
pip install -r requirements.txt
```

Проект не требует настройки внешних секретов или дополнительных системных библиотек. Все параметры хранятся в configs/ в открытом виде..

## 4. Как запустить проект

### 4.1. Запуск обучения модели (опционально, сервис запустится и без этого)

Обучение и эксперименты проводятся в Jupyter Notebook. Для воспроизведения результатов и переобучения финальной модели:

```bash
cd project
source .venv/bin/activate      # при необходимости
jupyter notebook notebooks/02_baseline_models.ipynb         
```

### 4.2. Запуск сервиса (API/веб-интерфейс)

```bash
cd project
source .venv/bin/activate      # при необходимости
uvicorn src.api.main:app --reload   # пример: FastAPI сервис
```

Также можно запустить через Docker:

```bash
cd project
docker build -t credit-scoring-api .
docker run -p 8000:8000 credit-scoring-api
```

Опишите:

- Сервис поднимается на порту 8000;
- Ключевые эндпоинты:
  - POST /api/v1/predict — предсказание кредитного риска (принимает параметры клиента, возвращает метку риска и вероятность).
  - GET /api/v1/health — проверка здоровья сервиса и статуса загрузки модели.
  - GET /api/v1/metrics — технические метрики сервиса (uptime, количество запросов, распределение предсказаний по категориям риска).

- Как протестировать: Откройте в браузере http://127.0.0.1:8000/docs (Swagger UI) или выполните запрос через curl:
  ```bash
  curl -X POST "http://127.0.0.1:8000/api/v1/predict" \
  -H "Content-Type: application/json" \
  -d '{"age": 35, "income": 75000, "employment_years": 10, "loan_amount": 25000, "credit_score": 720}'
  ```

## 6. Тесты 

В проекте реализованы автоматические тесты для проверки целостности данных и работоспособности API:

- **Sanity-check данных** (`tests/test_data.py`) – проверка отсутствия критических пропусков, корректности диапазонов признаков и баланса классов в датасете.
- **Юнит-тесты API** (`tests/test_api.py`) – проверка эндпоинтов `/`, `/health`, `/predict` на валидных и невалидных запросах, корректность схем валидации Pydantic и обработки ошибок сервера.
- В CI используется мокирование загрузки модели для изоляции тестов от объёмных артефактов.

**Запуск всех тестов:**

```bash
cd project
source .venv/bin/activate      # .venv\Scripts\activate для Windows
pytest tests/ -v
```

## 8. Ограничения и дальнейшая работа

> В текущей версии сервис использует статическую модель и работает с синтетическими данными. Метрики наблюдаемости собираются только в оперативной памяти приложения и сбрасываются при перезапуске.
> В дальнейшем планируется добавить:
>
> - Мониторинг дрейфа данных (Data Drift) для отслеживания изменения распределения входных данных.
> - Систему аутентификации и логирования запросов для безопасности API.
> - Хранение истории предсказаний в базе данных для последующего аудита.
